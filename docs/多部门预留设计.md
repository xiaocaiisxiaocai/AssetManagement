# 多部门预留设计（共享资产池）

**文档版本**：v1.0
**编写日期**：2026-06-11
**性质**：架构演进设计 / ADR
**关联文档**：需求文档、设计概要、架构设计文档

---

## 1. 背景与目标

当前系统是严格的**单部门**设计：架构定位为"单一部门内网 Web 应用"，资产无部门归属，`users.department` 只是自由文本（无受控实体），编号全局唯一，角色与报表均为全局口径。

业务方提出"可能要预留其他部门"。本设计的目标是把系统从**单租户演进为多租户（部门即租户）**，并在原型中演示多部门形态，同时把数据模型扩展点在设计文档中固化，为将来正式系统的平滑迁移铺路。

现阶段系统尚无真实数据、未上线，是预留的最低成本时机。

## 2. 关键决策（已与业务方确认）

| 决策项 | 结论 | 说明 |
|--------|------|------|
| 隔离模型 | **共享资产池** | 一套资产库，每个资产标记"归属部门"；允许跨部门借用/转让；报表可按部门汇总 |
| 转让与归属 | **归属随接收人转移** | 跨部门转让通过后，资产归属部门从转出部门划至接收人所在部门（人走资产走） |
| 跨部门审批人 | **申请人本部门审批** | 沿用现有审批人逻辑（申请人直属主管/代审人），跨部门不改变审批人——审批流零改动 |
| 资产编号 | **保持全局唯一** | 共享池中编号本就应唯一标识一个物理资产，跨部门转让编号不变；"按部门独立流水"列为未来可选项 |
| 权限隔离 | **本次仅演示，不做硬隔离** | 引入"部门级管理员"角色概念与按部门视图过滤用于演示；真正的后端权限隔离待正式上多部门时实现 |

### 2.1 一条需知晓的规则含义（风险提示 🟡）

"归属随转移 + 申请人本部门审批"组合下，B 部门员工可发起"将 A 部门资产转让给自己"并由 B 部门主管批准，即 **A 部门资产可被 B 部门单方批准拿走**。原型按此实现。若正式业务要求资产转出方有发言权，应改为"资产归属部门审批"或"双部门会签"——此项已在演进策略中记录为可切换点。

## 3. 数据模型扩展点

### 3.1 新增实体 `departments`

| 字段 | 类型 | 说明 |
|------|------|------|
| id | PK | 部门标识 |
| name | string | 部门名称 |
| code | string | 部门编码（唯一） |
| parent_id | FK，可空 | 支持部门层级（部门/子部门） |
| manager_id | FK→users | 部门负责人（部门级管理员） |
| status | enum | active / disabled |

### 3.2 既有表增加归属字段

- `assets` 增加 `department_id`（资产归属部门，非空，默认指向迁移时的"默认部门"）
- `users.department`（自由文本）升级为 `department_id` 外键 → `departments`
- `approval_flows` 转让工单增加 `from_department_id` / `to_department_id`，记录归属迁移轨迹
- 报表统计口径增加 department 维度

### 3.3 迁移策略

正式系统首次升级时：建 `departments` 表 → 插入一个"默认部门" → 存量 `assets`/`users` 的 `department_id` 全部回填为默认部门 → 之后再拆分。全过程是"加约束"而非"改地基"，无破坏性变更。

## 4. 本次原型演示范围（C 重演示）

可见 UI（改 `prototype/`）：

1. **顶栏部门切换器**："查看范围"下拉（全部部门 / 各部门），切换后资产列表与报表按所选部门**真过滤**重渲染——演示"按部门隔离视图"。
2. **资产列表**：列表视图增加"归属部门"列；筛选区增加"归属部门"下拉；列表按 `当前查看范围 + 筛选条件` 真实过滤，停留在列表视图。
3. **系统管理 → 新增"部门管理"页**：树形展示部门（含负责人、资产数、状态），支持新增/编辑/删除、添加子部门，复用通用编辑器。
4. **统计报表 → 资产汇总**：增加"按部门统计"表格（各部门资产数/在库/借出/价值/占比）。
5. **跨部门转让**：接收人下拉显示其部门；转让审批通过后**真实更新**该资产的保管人与归属部门，资产列表立即体现归属变化；审批详情与提示展示"归属：转出部门 → 接收部门"。
6. **部门级管理员**：在用户数据与部门管理页中体现"部门管理员"角色概念（负责人列）；顶栏说明部门管理员仅可见本部门。原型不做后端硬拦截。

数据模型预留 + 文档（第 3、5 节）三种演示深度都包含；本仓库选定 C。

## 5. 演进策略（未来正式上多部门时补齐）

- **权限硬隔离**：数据访问层按 `department_id` 强制过滤；部门级管理员仅能读写本部门资产；全局管理员可跨部门。
- **审批规则可切换**：将"申请人本部门审批"切换为"资产归属部门审批"或"双部门会签"（见 2.1）。
- **编号规则**：如需各部门独立流水，`number_rules` 增加 `department_id` 维度。
- **类别/位置共享策略**：明确类别树与位置树是全局共享还是各部门独立维护（当前按全局共享）。
- **跨部门调拨流程**：如需比转让更正式的资产调拨审批，可在共享池基础上扩展独立工单类型。

## 6. 测试

浏览器冒烟测试覆盖：部门切换真过滤、归属部门筛选、部门管理页增删改、按部门统计、跨部门转让的归属变更可见性、桌面与移动响应式布局。

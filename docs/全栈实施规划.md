# 部门资产管理系统 — 全栈实施规划

**版本**：v1.0 ｜ **日期**：2026-06-12 ｜ **性质**：项目实施总体规划（superpowers 规划产出）
**关联**：需求文档、设计概要、架构设计文档、多部门预留设计、审批工作流设计、资产层级编码设计、静态原型

---

## 1. 背景与目标

将现有「需求/设计文档 + 静态原型」实现为生产可用的全栈系统：前端采用 purest-admin 的 **vben** 版作为脚手架，后端按指定栈 **.NET 8 + ASP.NET Core Web API + EF Core + SQLite + Swagger UI** 自建。面向部门内网、80–100 人规模、单机部署。

## 2. 技术栈（已定）

| 层 | 选型 | 说明 |
|----|------|------|
| 前端脚手架 | purest-admin **vben** 版（Vue3 + Vite + vben-admin，带国际化） | 复用其 RBAC/菜单/权限路由/登录 |
| 后端框架 | ASP.NET Core Web API（.NET 8） | 分层架构，自建 |
| ORM | EF Core 8（Code First + Migrations） | — |
| 数据库 | SQLite（单文件） | 小规模单机；并发写有限但够用 |
| API 文档 | Swagger UI（Swashbuckle） | 开发期联调 |
| 认证 | JWT（Bearer） | 对接 vben 登录/权限 |
| 部署 | 内网单机（Kestrel/IIS + 前端静态托管） | 无定时任务、无邮件 |

## 3. 关键决策（含本次默认假设，待确认）

1. **后端自建，不沿用 purest-admin 后端** 🟢：purest-admin 后端是 SqlSugar + .NET 10 + MySQL，与指定的 EF Core + SQLite + .NET 8 不可调和；自建是唯一满足要求的路径。
2. **前端采用 vben 脚手架** 🟢：在其上开发四大业务模块页面，对接自建后端。
3. **自建后端提供 vben 期望的 API 契约** 🟡：登录、用户信息、动态菜单、按钮权限等接口需匹配 vben 前端约定（详见 §6），使前端改动最小。
4. **完整 RBAC** 🟡（默认）：用户/角色/权限/菜单 + 部门级管理员，vben 开箱即用其权限体系。
5. **沿用前期设计决策**：共享资产池多部门、审批工作流引擎（会签/或签/条件/加签/转签）、资产分类编码树（逐层拼接）、**移除定时任务与邮件**（超时仅标记、催办走站内）。

## 4. 后端架构（分层解决方案）

```
AssetManagement.sln
├── AssetManagement.Domain          # 实体、枚举、领域服务接口、领域规则
├── AssetManagement.Application      # 应用服务、DTO、用例编排、校验、AutoMapper
├── AssetManagement.Infrastructure   # EF Core DbContext + SQLite、仓储、JWT、审计拦截器、编号/工作流引擎实现
└── AssetManagement.Api              # Controllers、中间件、全局异常、Swagger、依赖注入
```
- **统一响应**：`{ code, message, data }`；统一异常中间件；统一分页。
- **审计**：ActionFilter 切面自动记录写操作（沿用设计概要 AOP 方案）。
- **编号引擎**：分类编码树逐层拼接 + 末层流水号（资产层级编码设计）。
- **工作流引擎**：流程定义/实例/节点流转（审批工作流设计），运行时会签/或签/条件/加签/转签。

## 5. 数据模型（EF Core Code First → SQLite）

核心实体（来自设计概要与各专项设计）：
- **RBAC**：`users, roles, permissions, menus, user_roles, role_permissions, role_menus`
- **组织**：`departments`（自引用，任意层级，code/manager/status）
- **分类编码树**：`asset_categories`（自引用，code_seg/code/full_path）
- **资产**：`assets`（asset_no, category_id, department_id, location_id, status, custodian_id, price…）、`locations`
- **审批工作流**：`workflows`(流程定义 + nodes JSON)、`approval_flows`(实例 + nodes 快照/current_node_index)、`flow_records`
- **系统**：`audit_logs`、`system_settings`
- 初始数据：种子（默认部门、管理员、角色权限、三套审批流、示例分类树）。

## 6. 前后端对接契约（vben ↔ 自建后端）

vben 前端依赖以下后端接口（自建后端须实现以匹配）：
- `POST /api/auth/login` → `{ token }`
- `GET /api/auth/user-info` → 用户基本信息 + 角色/权限码
- `GET /api/menu/routes`（或等价）→ 动态路由/菜单树（vben 据此生成左侧菜单与权限路由）
- 按钮级权限码（vben `v-access`/指令）
- 业务 RESTful 接口统一前缀 `/api/...`，Swagger 暴露。

> **风险**：vben 默认对接 purest-admin 的 SqlSugar 后端，其登录/菜单/权限响应结构需在 M1 对齐——要么后端按 vben 约定输出，要么前端改 `api` 层适配。M1 第一件事就是锁定该契约。

## 7. 里程碑

| 里程碑 | 范围 | 产出/验收 |
|--------|------|-----------|
| **M0 脚手架** | 后端四层解决方案 + EF Core/SQLite + Swagger 跑通；前端拉取 vben 版本地跑通 | 后端 `/swagger` 可访问、健康检查通过；前端登录页可渲染 |
| **M1 认证 + RBAC** | JWT 登录、用户/角色/权限/菜单 CRUD、动态菜单契约、vben 权限路由对接、审计切面 | vben 用真实账号登录、按权限渲染菜单、写操作进审计 |
| **M2 基础数据** | 组织架构树、资产分类编码树、位置、系统参数 | 四个管理页前后端打通，编码逐层拼接生效 |
| **M3 资产管理** | 资产 CRUD、编号自动生成、列表筛选/分页、导入导出（EPPlus） | 资产全生命周期基础、Excel 导入预校验 |
| **M4 审批工作流** | 流程设计器（前端）、工作流引擎（后端）、借用/归还/转让、加签/转签/会签/或签/条件 | 三类申请按可配置流程流转并执行业务动作 |
| **M5 报表 + 审计** | 资产汇总（按类别/部门）、借用明细、逾期、审计日志查询导出 | 报表数据正确、可导出 Excel |
| **M6 联调 + 部署** | 全链路联调、内网部署脚本、初始化与备份说明 | 内网可访问、SQLite 备份方案 |

**依赖顺序**：M0 → M1 →（M2、M3 可并行起步）→ M4 → M5 → M6。M4 工作流最复杂，置于基础数据与资产之后。

## 8. 第一阶段（M0 + M1）任务大纲

**M0 脚手架**
1. 建 `AssetManagement.sln` 四层项目；引用关系 Api→Application→Domain，Infrastructure→Domain。
2. Infrastructure：`AppDbContext`（EF Core + `Microsoft.EntityFrameworkCore.Sqlite`），连接字符串、首个迁移、启动时 `Migrate()`。
3. Api：Swashbuckle/Swagger UI、统一响应包装、全局异常中间件、健康检查 `/health`。
4. 前端：克隆 purest-admin vben 版，配置 `.env` 指向后端，`pnpm dev` 跑通登录壳。

**M1 认证 + RBAC**
1. Domain/Infra：RBAC 实体 + 迁移 + 种子（admin/角色/权限/菜单）。
2. JWT：登录签发、鉴权中间件、`[Authorize]` + 权限策略。
3. Application/Api：`auth/login`、`auth/user-info`、`menu/routes`、用户/角色/权限/菜单 CRUD。
4. 对接 vben：按 §6 契约输出，前端 `api` 层联通，权限路由 + 按钮权限生效。
5. 审计切面接入。

## 9. 风险与取舍

- **vben 契约适配**（M1 关键）：先锁定登录/菜单/权限响应结构。
- **EF Core + SQLite 并发写**：小规模可接受；导入等批量操作走事务、避免长写锁。
- **工作流引擎复杂度**：流程定义用 JSON 列存储节点，运行时实例克隆快照（与原型一致），降低关系建模复杂度。
- **前端业务页工作量**：四大模块 + 工作流设计器是主要工作量，M2–M4 为重头。

## 10. 测试与验收

- 后端：xUnit 单元测试（领域规则：编号拼接、工作流流转判定）+ 集成测试（关键 API）。
- 前端：关键流程手动冒烟（沿用原型测试清单）+ 可选 Playwright e2e。
- 每个里程碑有明确验收标准（§7）；联调以 Swagger + vben 真实对接为准。

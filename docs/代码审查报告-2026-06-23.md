### 1️⃣ 架构与设计 - ⭐⭐⭐⭐⭐ (5/5)

#### ✅ 优势

**1.1 DDD 四层架构清晰**
```
Domain → Application → Infrastructure → Api
依赖方向单向,职责边界明确
```
- Domain 层纯粹(无 EF 污染,纯函数工作流引擎)
- Application 层仅定义契约(DTO + 接口)
- Infrastructure 层聚合实现(EF、JWT、审计、文件存储)
- Api 层瘦控制器(每个 action 一行调用)

**1.2 BPMN 2.0 工作流引擎**(亮点 ⭐⭐⭐⭐⭐)
```csharp
// 纯函数设计,易测试
public static class BpmnEngine {
    public static void Start(IBpmnFlowInstance flow, BpmnProcess process)
    public static void Approve(...) 
    public static void Reject(...)
}
```
- Token 驱动并行流程执行
- 接口解耦(IBpmnFlowInstance 避免 Domain 依赖实体)
- 标准兼容(Camunda 扩展属性)
- 支持 UserTask、网关(排他/并行/包容)、条件表达式

**1.3 服务粒度合理**
- 按限界上下文划分(非每实体一个服务)
- IAssetService(CRUD + 详情/流转 + Excel 导入)
- IBaseDataService(部门/分类/位置共用)
- IWorkflowService(工作流 + 审批)

#### ⚠️ 需要改进

**1.4 缺少缓存层**
- 部门树/分类树/位置树每次查询都打数据库
- 建议: 引入内存缓存(IMemoryCache),TTL 5 分钟

**1.5 缺少领域事件**
- 资产状态变更、审批通过等关键事件散落在服务层
- 建议: 引入 MediatR + 领域事件,解耦副作用

### 3️⃣ 性能与可扩展性 - ⭐⭐⭐☆☆ (3.5/5)

#### ✅ 优势

**3.1 分页与索引**
```csharp
var pageSize = Math.Clamp(query.PageSize, 1, 200); // 限制最大 200
```
- ✅ 强制分页,避免一次性加载大数据集
- ✅ 关键字段建立索引(AssetNo、CategoryId、DepartmentId、Status)

**3.2 并发冲突重试**
```csharp
for (var attempt = 0; ; attempt++) {
    try {
        await _db.SaveChangesAsync();
        return await GetAsync(asset.Id);
    }
    catch (DbUpdateException) when (attempt < 3) {
        _db.Entry(asset).State = EntityState.Detached;
    }
}
```
✅ 资产编号唯一索引冲突自动重试(最多 3 次)

**3.3 批量导入事务保护**
```csharp
await using var tx = await _db.Database.BeginTransactionAsync();
// ...批量插入
await _db.SaveChangesAsync();
await tx.CommitAsync();
```
✅ 整批一个事务,失败整体回滚

#### ⚠️ 需要改进

**3.4 N+1 查询风险**
```csharp
// AssetService.ToDtos 每次查询都批量加载关联表
var categories = await _db.AssetCategories.Where(x => categoryIds.Contains(x.Id)).ToDictionaryAsync();
var departments = await _db.Departments.Where(x => departmentIds.Contains(x.Id)).ToDictionaryAsync();
```
- ✅ 已优化: 批量查询字典,非逐条查询
- ⚠️ 但首次查询 assets 时未 Include,需两次往返

**建议**:
```csharp
var assets = await _db.Assets
    .Include(x => x.Category)
    .Include(x => x.Department)
    .Include(x => x.Location)
    .ToListAsync();
```

**3.5 部门树递归查询低效**
```csharp
private int[] DescendantDepartmentIds(int rootId) {
    var departments = _db.Departments.AsNoTracking().Select(...).ToList();
    // 内存中递归遍历
}
```
- ⚠️ 每次都全表加载部门
- **建议**: 引入 IMemoryCache 缓存部门树(TTL 5 分钟)

**3.6 SQLite 并发写入限制** ⚠️ **关键**
- SQLite 仅支持一个写入事务(database-level lock)
- 高并发场景(>5 写/秒)会出现 `SQLITE_BUSY`
- **风险**: 资产导入(批量插入)、审批流程(并发审批)可能阻塞

**建议**:
1. 短期: 在 SQLite 连接字符串加 `Pooling=True;Cache=Shared`
2. 中期: 监控写入 QPS,<5 可接受
3. 长期: 用户超 100 人时迁移到 PostgreSQL/MySQL

**3.7 缺少查询超时配置**
```csharp
.UseSqlite(builder.Configuration.GetConnectionString("Default"))
```
- ⚠️ 未设置 CommandTimeout,默认 30 秒
- **建议**: 加 `.UseQueryTrackingBehavior(QueryTrackingBehavior.NoTracking)` 提升只读查询性能

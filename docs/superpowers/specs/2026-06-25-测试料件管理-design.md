# 测试料件管理 — 设计文档

- **日期**:2026-06-25
- **状态**:已与需求方确认,待评审
- **所属仓库**:部门资产管理系统（`backend/` + `web/`）
- **架构方案**:方案 A —— 独立限界上下文 + 复用横切设施

---

## 1. 背景与目标

### 1.1 业务背景

测试料件是**厂商寄来供测试使用的物料**，归属于某个测试项目，**不属于固定资产**。它与固定资产的关键差异：

- 编号只需临时编号，不需要像资产清单那样分三层并自动编号。
- 生命周期简单：只有「在用」「已退回厂商」两态，无报废、无维修。
- 可能在人员之间流转（甲 → 乙），流转**可能需要审批**（由全局开关控制）。

### 1.2 核心约束（需求方明确）

1. **不与固定资产共用**：独立的数据表、独立的菜单入口、独立的权限码。测试料件与固定资产的数据**不能混在一起**。
2. **这是新入口**：不挂在资产管理菜单下，新增一级菜单「测试料件」。
3. 复用底层成熟设施（BPMN 引擎、照片上传、RBAC、软删除、基础数据下拉），但**数据隔离**。

### 1.3 成功标准

- 测试料件的清单、详情、流转、审批、删除等操作全程不触碰 `Asset` / `ApprovalFlow` 等资产侧数据，零回归。
- 料件可按所属测试项目筛选与汇总。
- 流转审批开关开启时走 BPMN 审批流，关闭时直接转移生效。
- 后端新增测试全部通过，现有资产/审批测试不受影响。

## 2. 需求决策汇总（已确认）

| 议题 | 决策 |
|---|---|
| 临时编号 | 系统生成简单流水号 `TM-yyyyMMdd-NNN`（当日序号） |
| 所属项目 | 独立的「测试项目」基础数据表（可增删改，一项目挂多料件） |
| 流转审批 | 复用 BPMN 工作流引擎（`BpmnEngine` 纯函数），但用**独立流转实例表** |
| 流转粒度 | 整批转移（不拆分数量） |
| 生命周期状态 | 仅「在用 / 已退回厂商」两态（无报废、无维修） |
| 厂商字段 | 需要「厂商 / 来源」字段 |
| 预计退还日期 | 不需要 |
| 删除模型 | 复用资产软删除模型（软删 + 撤销 + 彻底删除三态） |
| 审批触发 | 全局开关控制（系统参数 `material.transfer.approval.enabled`，默认关闭） |
| 部门数据隔离 | **需要**，比照 `AssetService.ApplyQuery`（超管全部、部门管理员本部门+子部门） |
| 流转记录表 | 独立 `MaterialFlowRecord` 表（不复用 `FlowRecord`） |

## 3. 架构方案对比

### 方案 A（采纳）—— 独立限界上下文 + 复用横切设施

- 新建独立表 `TestMaterial` / `TestProject` / `MaterialFlow` / `MaterialFlowRecord`。
- 复用但不共享数据：`BpmnEngine`、`Workflow` 模板、`FileStorageService`、软删除三态、RBAC/审计、部门/位置/用户下拉。
- 独立权限码族 `material:*` / `project:manage`，独立一级菜单「测试料件」。
- **优点**：零回归（完全不碰资产表与 `ApprovalFlow`）；契合「不能混」；可独立测试。
- **代价**：少量重复脚手架（Service/Controller/前端页面）。

### 方案 B（否决）—— 泛化 `ApprovalFlow` 为通用审批表

- 去掉 `AssetId/AssetNo/AssetName`，改 `TargetType/TargetId`，资产与料件共用一张流转表。
- **否决理由**：要改动现有资产审批，回归风险高；共用一张表违背「数据不能混」。

### 方案 C（否决）—— 料件挂资产表加 `IsTestMaterial` 标志位

- **否决理由**：需求方明确拒绝「共用资产管理」。

## 4. 数据模型（新增 4 张独立表）

### 4.1 `TestMaterial`（测试料件）

| 字段 | 类型 | 说明 |
|---|---|---|
| Id | int | 主键 |
| MaterialNo | string | 系统生成临时编号 `TM-yyyyMMdd-NNN` |
| Name | string | 料件名称 |
| ProjectId | int | 所属测试项目（外键 → `TestProject`） |
| VendorName | string? | 厂商 / 来源（寄件方） |
| Model | string? | 型号 |
| Brand | string? | 品牌 |
| Quantity | int | 数量（整批，默认 1） |
| DepartmentId | int? | 归属部门 |
| LocationId | int? | 存放位置 |
| CustodianId | int? | 当前保管人（流转即改此字段） |
| ReceivedDate | DateTime? | 接收日期 |
| Status | enum `MaterialStatus` | `InUse=0` 在用 / `ReturnedToVendor=1` 已退回厂商 |
| ImageUrls | string? | 照片 URL 逗号分隔（复用 `/api/files`，最多 5 张） |
| Remark | string? | 备注 |
| CreatedAt | DateTime | 创建时间 |
| IsDeleted | bool | 软删除标志 |
| DeletedAt | DateTime? | 删除时间 |

### 4.2 `TestProject`（测试项目基础数据）

| 字段 | 类型 | 说明 |
|---|---|---|
| Id | int | 主键 |
| Name | string | 项目名称 |
| Code | string? | 项目编号（可选，便于检索） |
| Description | string? | 描述 |

| IsDeleted | bool | 软删除标志 |

> 一个项目挂多个料件；料件清单可按 `ProjectId` 筛选。删除项目前若仍有未删除料件，禁止删除（比照「有资产的分类不可删」）。

### 4.3 `MaterialFlow`（料件流转单，实现 `IBpmnFlowInstance`）

结构对标 `ApprovalFlow`，但**独立表、独立字段**：

| 字段 | 类型 | 说明 |
|---|---|---|
| Id | int | 主键 |

| BizType | string | 固定 `material_transfer` |
| WorkflowId | int | 关联的 BPMN 工作流模板 |
| MaterialId | int | 料件 ID |
| MaterialNo | string | 料件编号（冗余，便于展示） |
| MaterialName | string | 料件名称（冗余） |
| ApplicantId | int | 发起人（当前保管人） |
| Applicant | string | 发起人姓名 |
| ApplicantDept | string? | 发起人部门 |
| TransfereeId | int? | 受让人 ID |
| Transferee | string? | 受让人姓名 |
| TransfereeDept | string? | 受让人部门 |
| Reason | string? | 转移原因 |
| Status | string | `pending` / `approved` / `rejected` |
| CurrentNodeIds | List\<string\> | BPMN 活跃节点（复用 `IBpmnFlowInstance`） |
| BpmnTokens | Dictionary\<string, BpmnToken\> | Token 状态字典 |
| ApplyTime | DateTime | 发起时间 |
| Deadline | DateTime | 截止时间 |

### 4.4 `MaterialFlowRecord`（流转操作记录，独立表）

| 字段 | 类型 | 说明 |
|---|---|---|
| Id | int | 主键 |
| FlowId | int | 关联 `MaterialFlow.Id` |
| Action | string | 操作（发起/通过/驳回/直接转移） |
| Operator | string? | 操作人 |
| Comment | string? | 备注 |
| OperatedAt | DateTime | 操作时间 |

### 4.5 编号生成

新建领域服务 `MaterialNoGenerator`（对标 `AssetNoGenerator`）：
- 格式 `TM-yyyyMMdd-NNN`，`NNN` 为当日已建料件的递增序号（从 001 开始，按当日 `MaterialNo` 前缀计数）。
- 纯领域逻辑，有独立单元测试。

## 5. 后端分层落地（DDD 四层）

依赖方向 Api → Infrastructure → Application → Domain，遵循现有约定。

### 5.1 Domain（`AssetManagement.Domain`）

- 实体：`TestMaterial`、`TestProject`、`MaterialFlow`、`MaterialFlowRecord`。
- 枚举：`MaterialStatus { InUse=0, ReturnedToVendor=1 }`。
- 领域服务：`MaterialNoGenerator`。
- `MaterialFlow` 实现 `IBpmnFlowInstance`，复用 `Domain/Workflow/BpmnEngine`（不新增引擎）。

### 5.2 Application（`AssetManagement.Application`）

按限界上下文粗粒度划分服务接口 + DTO（复用 `Common/ApiResult`、`PagedResult`、`BizException`）：

- **`ITestMaterialService`**：料件 CRUD、分页查询（含部门隔离 + 删除三态 + 按项目筛选）、详情（含流转时间线、允许查看已删除）、软删除/撤销/彻底删除、发起转移（`InitiateTransferAsync`）、退回厂商（改状态）。
- **`ITestProjectService`**：项目 CRUD（软删除，删除前校验无关联料件）。
- **`IMaterialFlowService`**：流转发起（内部判断全局开关）、审批通过、驳回、待我审批查询、我发起的查询、流转详情。

DTO：`TestMaterialDto` / `TestMaterialDetailDto` / `SaveTestMaterialRequest` / `TestMaterialQuery` / `TestProjectDto` / `SaveTestProjectRequest` / `MaterialFlowDto` / `InitiateTransferRequest` / `MaterialApprovalRequest` 等。

### 5.3 Infrastructure（`AssetManagement.Infrastructure`）

- 服务实现：`TestMaterialService`、`TestProjectService`、`MaterialFlowService`。
- 持久化：4 个 `IEntityTypeConfiguration`（每实体一个）+ `AppDbContext` 加 `DbSet`。
- **部门隔离**：`TestMaterialService` 注入 `IHttpContextAccessor`，新建 `ApplyDepartmentScope` 方法（逻辑比照 `AssetService.ApplyQuery`）——超管不限制；部门管理员（`dept_admin` 非 `admin`）过滤本部门+子部门；普通员工按现有资产口径处理（保持一致）。
- **流转副作用**：流程完成时由 `MaterialFlowService` 直接落地（改 `TestMaterial.CustodianId` 为受让人），**不复用** `IBizEffectApplier`（它绑死 `ApprovalFlow`）。
- 迁移：`AddTestMaterial`（在 `backend\` 目录执行 `dotnet ef migrations add`）。
- 种子：`DbSeeder` 增量幂等注册菜单、权限、`material_transfer` 默认工作流模板、系统参数（详见第 7 节）。

### 5.4 Api（`AssetManagement.Api`）

瘦控制器，每 action 一行调用对应 Service：

- `TestMaterialController`（路由 `api/test-materials`）：列表、详情、增改、删除/撤销/彻底删除、发起转移、退回厂商。
- `TestProjectController`（路由 `api/test-projects`）：项目 CRUD。
- `MaterialFlowController`（路由 `api/material-flows`）：待我审批、我的发起、流转详情、通过、驳回。

所有 action 用 `[HasPermission("material:xxx")]` 标注。

## 6. 流转 + 审批流程（全局开关控制）

1. 保管人在料件详情点「转移」→ 选受让人 + 填原因 → 提交（`InitiateTransferAsync`）。
2. `MaterialFlowService` 读系统参数 `material.transfer.approval.enabled`：
   - **关闭（默认）**：直接将 `CustodianId` 改为受让人，写一条 `MaterialFlowRecord`（Action=直接转移），不生成审批单。
   - **开启**：调 `BpmnEngine.Start()` 生成 `MaterialFlow` 审批单，Token 推进到第一个 UserTask；料件被「进行中流转单」锁定（不可重复发起转移、不可删除）。
3. 审批人通过 → `BpmnEngine.Approve()` 推进 Token；到达 EndEvent → 流程完成 → 落地副作用（改 `CustodianId`）→ 料件解锁。
4. 审批驳回 → `MaterialFlow.Status=rejected`，料件保管人不变、解锁。

**状态说明**：料件显式状态仍只有「在用 / 已退回厂商」两态；「流转审批中」是**派生锁定标志**（查询该料件是否存在 `Status=pending` 的 `MaterialFlow`），不进入 `MaterialStatus` 枚举。

**审批人配置**：复用 BPMN 设计器与 `Workflow` 模板表，种子放一个 `BizType="material_transfer"` 的默认工作流（发起 → 单节点审批 → 结束）的 BPMN XML。后续可在设计器中调整审批人/网关。

## 7. 权限与菜单种子（`DbSeeder` 增量、幂等）

### 7.1 权限码族

`material:view` / `material:create` / `material:edit` / `material:delete` / `material:restore` / `material:purge` / `material:transfer` / `material:approve` / `project:manage`。

默认角色授权：
- `admin`：全部。
- `dept_admin`：`view` / `create` / `edit` / `transfer` / `restore` / `approve` / `project:manage`（不含 `purge`）。
- 普通员工：至少 `material:view`（具体按后续需求微调）。

### 7.2 菜单（一级入口「测试料件」）

| 子菜单 | Path | Component | 权限 |
|---|---|---|---|
| 料件清单 | `/material/list` | `/material/list/index` | `material:view` |
| 测试项目 | `/material/projects` | `/material/projects/index` | `project:manage` |
| 流转/审批 | `/material/transfers` | `/material/transfers/index` | `material:transfer` |

### 7.3 系统参数

`material.transfer.approval.enabled`，默认 `false`（关闭审批，直接转移）。在 `views/admin/settings` 中可配置。

### 7.4 默认工作流模板

`material_transfer` 一条默认 BPMN XML 模板（发起→审批→结束），增量种子写入 `Workflow` 表。

## 8. 前端（独立一级入口，apps/web-ele）

- `views/material/list` — 料件清单：表格 + 查询（编号/名称/项目/部门/状态/删除三态）；行操作：详情、编辑、转移、删除/撤销/彻底删除（按 `material:*` 权限显隐）。
- `views/material/projects` — 测试项目管理（CRUD）。
- `views/material/transfers` — 流转/审批（待我审批 + 我的发起；开关开启时有意义）。独立，不并入资产审批中心。
- `views/material/components/MaterialDetailDialog.vue` — 详情对话框 + 流转时间线，对标 `AssetDetailDialog.vue`（`ElDescriptions` 展示，高亮删除/退回状态）。
- `api/material.ts` + `api/test-project.ts`，风格对标 `api/asset.ts`（`requestClient` 封装、`ApiResult` / `PagedResult` 类型）。
- 部门 / 位置 / 保管人下拉**复用** `api/base-data.ts` 现有 options 接口，不重复造。
- 照片上传复用现有 `/api/files` 上传组件。
- 提交前跑 `pnpm -F @vben/web-ele run typecheck`。

## 11. 风险与边界

- **风险**：BPMN 默认模板的审批人配置若未设好，开关开启时流程可能卡住 → 种子提供可用的默认模板，并在文档注明。
- **边界**：本期不做数量拆分流转、不做预计退还日期/逾期提醒、不做料件 Excel 批量导入（后续如需可比照资产 `AssetImportController` 扩展）。
- **部门隔离**口径与资产保持一致；若测试料件实际为跨部门共享，再行调整。

## 1. 背景与目标

当前审批是**固定流程**：申请 → 直属主管/代审人审批 → 通过/驳回，审批人由"直属主管"规则写死，并依赖定时任务（超时自动审批、逾期提醒）与邮件通知。

业务方要求审批"像用友那样自由设计"——可自由加节点、加签、会签/或签、按条件分支。本设计把固定审批升级为**可配置的工作流引擎 + 流程图式设计器**，并**移除定时任务与邮件提醒**。

## 2. 范围边界（已与业务方确认）

**原型实现**：流程图式可视化设计器、节点类型（发起/审批/会签/或签/条件/抄送/结束）、审批人来源（指定人/角色/直属主管/部门负责人/发起人自选）、借用/转让/归还各一套可编辑流程、运行时按定义流转（时间线）、审批操作（通过/驳回/加签/转签）、会签全通过推进、或签一人推进、条件分支按金额走不同路径。

**列入演进（原型不做）**：并行网关汇聚、子流程、流程版本管理、表单设计器、流程模拟器。

## 3. 数据模型

### 3.1 流程定义 `workflows`
```
{
  id, name, bizType(borrow/transfer/return),
  nodes: [
    { id, name, type, approverType, approver, signers[], condition }
  ]
}
```
- **节点类型 type**：`start` 发起 / `approval` 单人审批 / `countersign` 会签（所有人通过）/ `orsign` 或签（一人通过）/ `condition` 条件分支 / `cc` 抄送 / `end` 结束
- **审批人来源 approverType**：`user` 指定人 / `role` 角色 / `supervisor` 直属主管（解析时优先有效代审人）/ `deptManager` 部门负责人 / `applicantPick` 发起人指定
- **condition**：条件节点的进入条件（如 `金额 > 5000`），不满足则跳过该节点

### 3.2 审批实例（工单升级）
在原 `approvals` 工单上增加：
```
workflowId, amount, currentNodeIndex,
nodes: [
  { name, type, status(done/current/pending/skipped), approver, opinion, time,
    signers[], signStates{}, addedSigners[] }
]
```

## 4. 流程设计器（系统管理 → 审批流程）

- 流程图式纵向：节点卡片 + 连线箭头 + 条件分支双路（是/否）
- 顶部切换流程（借用/转让/归还）
- `[+ 添加节点]`：选节点类型插入；点击节点弹出配置（节点名、类型、审批人来源、会签/或签成员、条件表达式）
- 节点操作：上移 / 下移 / 删除（发起与结束节点固定）
- 保存流程（原型写入内存 `mockData.workflows`）

## 5. 运行时

- **发起**：按 bizType 匹配流程模板，克隆 nodes 生成实例，首个审批节点置 `current`
- **时间线**：审批详情按节点展示——已办 ✓（审批人+意见+时间）、当前 ⏳、待办 ○、跳过 ⊘
- **审批操作**：
  - **通过**：`approval` → 节点 done，推进；`countersign` 会签 → 记录该人通过，`signStates` 全通过才 done 推进；`orsign` 或签 → 一人通过即 done 推进；`condition` → 按 `amount` 判定，满足则审批、不满足置 `skipped` 直接推进
  - **驳回**：工单 `rejected`，流程终止
  - **加签**：当前节点 `addedSigners` 增加审批人（前加签=需先于原审批人、后加签=原审批人之后），加签人全部通过才算节点通过
  - **转签**：当前节点 `approver` 改为他人
  - **抄送 cc**：自动通过并记录知会（站内，无邮件）
- **完成**：所有节点 done → 工单 `approved`，执行业务动作（借出 / 转让归属转移 / 归还入库）

## 6. 移除定时任务与邮件

- 删除：超时自动审批、逾期提醒定时任务；全部邮件通知；系统参数中的 SMTP、审批超时小时数、逾期催还间隔等配置项
- **超时**：保留"审批截止"展示与列表"已超时"标记，但不自动处理（纯人工）
- **代审设置**：并入引擎——`supervisor` 类节点解析审批人时优先取有效代审人；不再作为独立定时逻辑
- **逾期资产报表**：保留查看，催还由"邮件"改为"站内手动催办"

## 9. BPMN 2.0 实现（2026-06-22 新增）

### 9.1 架构升级

从简单的节点数组模型升级到标准 BPMN 2.0 工作流引擎：

**旧模型**：
```javascript
{
  nodes: [
    { id, name, type: 'approval', approver: '张三' },
    { id, name, type: 'countersign', signers: ['李四', '王五'] }
  ],
  currentNodeIndex: 1
}
```

**新模型（BPMN 2.0）**：
```xml
<bpmn:process id="Process_borrow">
  <bpmn:userTask id="Task_1" name="审批" camunda:assignee="supervisor" />
  <bpmn:exclusiveGateway id="Gateway_1" />
  <bpmn:parallelGateway id="Gateway_2" />
</bpmn:process>
```

### 9.2 核心组件

#### 后端

| 组件 | 文件 | 功能 |
|------|------|------|
| BPMN 解析器 | `Domain/Workflow/BpmnParser.cs` | 解析 BPMN XML，构建流程图 |
| BPMN 引擎 | `Domain/Workflow/BpmnEngine.cs` | Token 驱动执行引擎 |
| 工作流服务 | `Infrastructure/Workflow/WorkflowService.cs` | 编排层，处理审批人解析和业务副作用 |
| 实体 | `Workflow.BpmnXml`、`ApprovalFlow.BpmnTokens` | 存储 BPMN 定义和运行时状态 |

#### 前端

| 组件 | 文件 | 功能 |
|------|------|------|
| 设计器 | `views/admin/workflows/bpmn-modeler.vue` | bpmn-js 可视化建模 |
| 属性面板 | `views/admin/workflows/bpmn-properties.vue` | 配置审批人、条件表达式 |
| 列表页 | `views/admin/workflows/index.vue` | 管理工作流定义 |
| API | `api/workflow.ts` | 类型定义和接口封装 |

### 9.3 支持的 BPMN 元素

| 元素 | 说明 | 用途 |
|------|------|------|
| StartEvent | 开始事件 | 流程入口 |
| EndEvent | 结束事件 | 流程出口 |
| UserTask | 用户任务 | 审批节点 |
| ServiceTask | 服务任务 | 自动任务（预留） |
| ExclusiveGateway | 排他网关 | 条件分支（单选一条路径） |
| ParallelGateway | 并行网关 | 并行执行（所有路径同时执行） |
| InclusiveGateway | 包容网关 | 多条件分支（满足条件的都执行） |
| SequenceFlow | 连线 | 流程流向，可配置条件表达式 |

### 9.4 审批人配置

使用 Camunda 扩展属性 `camunda:assignee` 配置审批人：

| 值 | 说明 | 解析逻辑 |
|---|------|---------|
| `supervisor` | 直属主管 | 自动解析为申请人的上级 |
| `deptManager` | 部门经理 | 自动解析为申请人所在部门的管理员 |
| `{userId}` | 用户 ID | 指定具体用户（如 `1001`） |
| `{roleCode}` | 角色代码 | 角色下的所有用户（如 `admin`、`warehouse`） |

**示例**：
```xml
<bpmn:userTask id="Task_1" name="直属主管审批" 
               camunda:assignee="supervisor" />
```

### 9.5 条件表达式

在 SequenceFlow 上配置条件，支持简单表达式求值：

```xml
<bpmn:sequenceFlow id="Flow_high" sourceRef="Gateway_1" targetRef="Task_2">
  <bpmn:conditionExpression>${amount} > 5000</bpmn:conditionExpression>
</bpmn:sequenceFlow>
```

**支持的变量**：
- `${amount}` - 审批金额
- `${applicantDept}` - 申请人部门
- 更多变量可在 `BpmnEngine.EvaluateCondition()` 中扩展

### 9.6 Token 驱动模型

每个活跃节点有一个 Token，支持并行流程：

```csharp
// 流程实例运行时状态
flow.CurrentNodeIds = ["Task_1", "Task_2"];  // 两个节点同时活跃
flow.BpmnTokens = {
    "Task_1": { Status: Active, NodeId: "Task_1", NodeName: "主管审批" },
    "Task_2": { Status: Active, NodeId: "Task_2", NodeName: "经理审批" }
};
```

### 9.7 执行流程

1. **启动流程**：
   ```csharp
   var process = BpmnParser.Parse(workflow.BpmnXml);
   BpmnEngine.Start(flow, process);
   // Token 推进到第一个 UserTask
   ```

2. **审批推进**：
   ```csharp
   BpmnEngine.Approve(flow, process, nodeId, userId);
   // Token 根据流程定义推进
   // 遇到网关自动评估条件
   ```

3. **流程完成**：
   ```csharp
   if (flow.Status == "approved") {
       // 执行业务副作用（资产状态变更等）
   }
   ```

### 9.9 优势与限制

#### 优势
- ✅ **标准化**：使用国际标准 BPMN 2.0
- ✅ **灵活性**：支持复杂的网关和并行流程
- ✅ **可扩展**：易于添加新的 BPMN 元素
- ✅ **可视化**：专业的 bpmn-js 设计器

#### 当前限制
- ⚠️ **加签/转签**：BPMN 标准中无对应概念，暂不支持（可通过扩展实现）
